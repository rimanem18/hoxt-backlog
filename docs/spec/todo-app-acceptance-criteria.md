# TODOリストアプリ 受け入れ基準

## 📄 ドキュメント情報

- **作成日**: 2025-11-06
- **要件名**: TODO リストアプリ
- **バージョン**: 1.0.0
- **関連文書**: [todo-app-requirements.md](todo-app-requirements.md), [todo-app-user-stories.md](todo-app-user-stories.md)

## 概要

このドキュメントはTODOリストアプリ機能の受け入れ基準とテスト項目を記載します。
すべての機能要件・非機能要件を満たすことを確認するための具体的なテストケースを定義します。

## 機能テスト基準

**【信頼性レベル凡例】**:
- 🔵 **青信号**: ユーザーヒアリング・既存設計から確実なテスト基準
- 🟡 **黄信号**: 既存設計・一般的な慣習から妥当な推測
- 🔴 **赤信号**: 既存資料にない推測

### REQ-001: タスク作成機能の受け入れ基準 🔵 *ユーザーヒアリング 2025-11-06*

**Given(前提条件)**:
- ユーザーがログイン済み
- TODOリスト画面にアクセスしている

**When(実行条件)**:
- リスト上部の入力フォームにタイトル「明日の会議資料作成」を入力
- Enterキーを押下

**Then(期待結果)**:
- タスクが一覧の最上部に追加される
- タスクのタイトルが「明日の会議資料作成」である
- タスクの優先度がデフォルト値「中」である
- タスクのステータスがデフォルト値「未着手」である
- 入力フォームが自動的にクリアされる
- タスクのuser_idが現在のログインユーザーのIDである

**テストケース**:
- [x] 正常系: 有効なタイトルでタスクが作成される
- [x] 異常系: タイトルが空文字の場合、エラーメッセージ「タイトルを入力してください」が表示される
- [x] 境界値: タイトルが100文字の場合、正常に作成される
- [x] 境界値: タイトルが101文字の場合、エラーメッセージ「タイトルは100文字以内で入力してください」が表示される
- [x] セキュリティ: XSS攻撃を試みるタイトル(`<script>alert('XSS')</script>`)がエスケープされて保存される

### REQ-002: タスク更新機能の受け入れ基準 🔵 *ユーザーヒアリング 2025-11-06*

**Given(前提条件)**:
- ユーザーがログイン済み
- 既存のタスク「会議資料作成」が存在する

**When(実行条件)**:
- タスクアイテムをクリックしてモーダルを開く
- タイトルを「会議資料作成(完了)」に変更
- 説明に「## チェックリスト\n- [x] 資料作成\n- [ ] レビュー依頼」を入力
- 優先度を「高」に変更
- 保存ボタンをクリック

**Then(期待結果)**:
- モーダルが閉じる
- タスクのタイトルが「会議資料作成(完了)」に更新される
- タスクの説明がMarkdown形式で保存される
- タスクの優先度が「高」に更新される
- 一覧画面で優先度の視覚的表現(色・アイコン)が更新される
- 更新日時(updatedAt)が現在時刻に更新される

**テストケース**:
- [x] 正常系: タイトル・説明・優先度がすべて更新される
- [x] 正常系: タイトルのみ更新される
- [x] 正常系: 説明のみ更新される
- [x] 正常系: 優先度のみ更新される
- [x] 異常系: タイトルを空文字に変更しようとするとエラーになる
- [x] 異常系: 存在しないタスクIDで更新しようとすると404エラーになる
- [x] セキュリティ: 他ユーザーのタスクを更新しようとすると403エラーになる

### REQ-006: タスク一覧表示機能の受け入れ基準 🔵 *ユーザーヒアリング 2025-11-06*

**テストケース**:
- [x] 正常系: 自分のタスクのみが表示される
- [x] 正常系: デフォルトで作成日時(新しい順)でソートされている
- [x] 正常系: タスクが0件の場合、空状態メッセージが表示される
- [x] パフォーマンス: 100件のタスクでもスムーズに表示される
- [x] セキュリティ: 他ユーザーのタスクは表示されない(RLS)

**NFR-001: タスク一覧取得のパフォーマンス 🟡 *既存パフォーマンス要件からの推測***

**テストシナリオ**:
- 100件のタスクが存在する状態
- TODOリスト画面にアクセス
- レスポンス時間を計測

**合格基準**:
- [x] レスポンス時間が1秒以内である
- [x] フィルタ・ソート適用時も1秒以内である
- [x] 同時に10ユーザーがアクセスしても性能劣化しない

**テスト方法**:
- 負荷テストツール: Bunのベンチマークツール

- 合格基準: 平均レスポンス時間1秒以内

## テスト自動化

### ユニット/統合テスト(Bun test)

**バックエンド**:
```bash
docker compose exec server bun test
```

**フロントエンド**:
```bash
docker compose exec client bun test
```

**カバレッジ目標**: 80%以上

### E2Eテスト(Playwright)

```bash
docker compose exec e2e npx playwright test
```

**主要テストシナリオ**:
- ログイン → タスク追加 → 編集 → ステータス変更(未着手→進行中→完了) → 削除の一連の流れ
- フィルタ・ソート機能の動作確認(ステータスフィルタ複数選択含む)
- エラーハンドリングの確認

### セキュリティスキャン(Semgrep)

```bash
docker compose run --rm semgrep semgrep scan --config=auto
```

**合格基準**: 脆弱性0件
