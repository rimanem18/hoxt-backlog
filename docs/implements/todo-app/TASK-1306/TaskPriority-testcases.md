- **入力値**:
  - `priority1`: `'high'`
  - `priority2`: `'high'`（priority1 と同じ値）
  - `priority3`: `'medium'`（priority1 と異なる値）
  - **入力データの意味**: 等価性比較のテストケースパターン
  - **選択理由**: 同値・異値の両パターンを網羅

  - `priority1.equals(priority2)` が `true` を返す
  - `priority1.equals(priority3)` が `false` を返す
  - **期待結果の理由**: 値オブジェクトは値の等価性で判定されるべき（DDD原則）
- **テストの目的**: equals() メソッドの正常動作確認
  - **確認ポイント**:
    - 同じ値を持つインスタンスが等価と判定される
    - 異なる値を持つインスタンスが非等価と判定される
    - 値オブジェクトの基本原則（識別子ではなく値で判定）の遵守

**実装例**:
```typescript
test('値オブジェクトの等価性比較ができる', () => {
  // 【テスト目的】: equals()メソッドが値の等価性を正しく判定することを確認
  // 【テスト内容】: 同じ値を持つインスタンスと異なる値を持つインスタンスでequals()を実行
  // 【期待される動作】: 同値の場合true、異値の場合falseを返す

  // 【テストデータ準備】: 等価性比較用に3つのインスタンスを用意
```

### TC-009: undefinedでエラーがスローされる

🟡 *一般的なバリデーション要件、既存テストパターンより*

- **テスト名**: undefinedでエラーがスローされる
  - **エラーケースの概要**: undefined が渡された場合のバリデーションエラー
  - **エラー処理の重要性**: TypeScript の型システムだけでは防げない実行時のundefined値を検出
- **入力値**: `undefined`（型アサーション `as any` で渡す）
  - **不正な理由**: 優先度は string 型の必須項目であり、undefined は許可されない
  - **実際の発生シナリオ**: オプショナルパラメータが未指定の場合や、JavaScript からの呼び出し

  - エラーメッセージに `'不正な優先度です'` が含まれる
  - **エラーメッセージの内容**: 優先度が不正であることを明示
  - **システムの安全性**: 実行時の undefined 値を検出してシステムの安定性を保証
- **テストの目的**: undefined 値のバリデーション
  - **品質保証の観点**:
    - 実行時の型安全性確保
    - 未定義値エラーの防止
    - オプショナルパラメータへの対応

## 3. 境界値テストケース

🟡 *列挙型の性質上、境界値テストは不要*

TaskPriority は列挙型（`'high'`, `'medium'`, `'low'`）であり、数値的な境界値は存在しません。
正常系テストで3つの有効値すべてをカバーしているため、境界値テストケースは追加不要です。

**理由**:
- 有効値が離散的（3つのみ）
- 最小値・最大値という概念が存在しない
- TC-001〜TC-003 で全有効値を網羅済み
